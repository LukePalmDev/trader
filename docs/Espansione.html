04 — Analisi

Punti di forza del cambiamento

La transizione verso un'infrastruttura dedicata (scenari B o C) risolve strutturalmente i limiti architetturali dell'attuale setup e abilita funzionalità oggi impossibili.

✦ Forze — Infrastruttura
  • Uptime garantito 99.9%. I dati vengono raccolti anche quando il Mac è spento, in aggiornamento o in riparazione. Oggi 1 riavvio del Mac Mini = minimo 3 run Subito perse (ogni 6 ore).
  • Eliminazione del single point of failure. Il database non risiede più su un solo dispositivo fisico non ridondato.
  • Playwright in headless mode. Sui VPS Linux, Chromium gira in headless stabile — nessun problema con Xcode, screen server o sessioni utente attive.
  • Backup automatici. I VPS moderni offrono snapshot giornalieri integrati a €1–2/mese. Oggi il backup del DB dipende da TimeMachine locale.
  • Indipendenza da GitHub Actions. I workflow possono diventare semplici cron job su crontab/systemd, senza dipendere da quote, runner availability o workflow_run trigger.
✦ Forze — Usabilità
  • Accesso ovunque. Il viewer (oggi solo su 127.0.0.1:8080) diventa accessibile da iPhone, iPad, qualsiasi browser con autenticazione Bearer token — già implementata nel codice.
  • Mac App come shell nativa. WKWebView su macOS permette di incapsulare il viewer esistente in un'app nativa in poche ore, senza riscrivere il frontend. Notifiche native, menu bar, Spotlight integration.
  • Alert Telegram già funzionanti. L'infrastruttura di alerting è già pronta e non richiede modifiche — solo puntare a un host diverso.
  • API REST già pronta. I 30+ endpoint di server.py con auth Bearer sono già production-ready per essere esposti pubblicamente con un reverse proxy (nginx/caddy).
  • Eliminazione dipendenza dalla rete locale. Oggi il viewer non funziona da reti esterne; con HTTPS sul VPS si risolve con un dominio e un certificato Let's Encrypt gratuito.

05 — Analisi

Punti di debolezza e rischi

La migrazione introduce nuove classi di rischio che devono essere valutate prima della decisione. Di seguito i fattori critici per ogni dimensione.

◆ Criticità tecniche
  • Playwright su Linux headless. L'attuale scraper Subito usa playwright_channel = "chrome" (Chrome di sistema). Su un VPS Linux non esiste Chrome di sistema: richiede chromium bundled + dipendenze di sistema (libgbm, libnss, ecc.). Migrazione: 2–4 ore di test.
  • Subito scraper usa fingerprint macOS. Il user-agent attuale dichiara Macintosh; Intel Mac OS X 10_15_7. Su Linux va aggiornato per evitare incongruenze TLS/user-agent che possono aumentare la probabilità di ban.
  • SQLite multi-process. Il DB SQLite non è progettato per accesso multi-processo concorrente su rete. Se si vuole accedere al DB da più istanze contemporanee (es. scraper + API), occorre implementare WAL mode o migrare a PostgreSQL (scenario D).
  • Gestione secrets su VPS. Le credenziali (Telegram token, Anthropic API key) oggi sono GitHub Secrets. Su VPS vanno gestite manualmente via .env o secret manager.
◆ Criticità operative
  • Costo ricorrente. Anche il VPS più economico (€4.5/mese) introduce un costo fisso mensile che oggi non esiste. Su 24 mesi: €108–€216.
  • Manutenzione OS. Un VPS richiede aggiornamenti di sicurezza, monitoraggio dello spazio disco e gestione del processo di avvio dei servizi (systemd units). Stima: 1–2 ore/mese.
  • Mac App: complessità di sviluppo. Lo sviluppo di una Mac App nativa richiede competenze Swift/SwiftUI o Electron. Un wrapper WKWebView di base richiede 1–2 settimane; un'app menu-bar con notifiche native e auto-launch 3–4 settimane. Nessun componente attuale è Swift.
  • Sicurezza del viewer esposto. Esporre server.py su internet richiede obbligatoriamente: HTTPS (TLS), reverse proxy (nginx/caddy), rate limiting, e un audit degli endpoint POST. Il token Bearer attuale è vuoto di default in config.toml.
  • Continuità del Mac Mini. Durante la migrazione, il runner self-hosted sul Mac Mini deve continuare a operare fino al completo switch-over — doppio setup temporaneo.

06 — Condizioni

Quando conviene · Quando no

La decisione è contestuale. Le seguenti condizioni determinano se la migrazione è strategicamente vantaggiosa nel momento attuale.

Condizioni FAVOREVOLI alla migrazione
  • Il Mac Mini si spegne o va offline frequentemente.
    Ogni ora di downtime = almeno 1 run Subito persa su 491 già eseguite (0.2% dei dati). Su base annua, 10 downtime da 6h = 10 run perse = gap nei trend.
  • Si vuole monitorare da mobile o da reti esterne.
    L'alert Telegram notifica ma non permette di vedere i dati senza essere sulla rete locale. Con il VPS bastano dati mobili.
  • Il progetto è destinato a crescere oltre il mercato Xbox.
    Aggiungere PlayStation, Nintendo o altri mercati (Vinted, Wallapop) triplica il volume: il Mac Mini diventa collo di bottiglia.
  • Si prevede di condividere il tool con altri utenti.
    Un VPS con HTTPS rende il viewer condivisibile via link protetto a zero costo aggiuntivo.
  • Il budget mensile per il progetto è ≥ €5.
    Hetzner CX22 a €4.51/mese è il VPS con il miglior rapporto performance/costo disponibile in Europa.
Condizioni SFAVOREVOLI alla migrazione
  • Il Mac Mini è sempre acceso e stabile.
    Se l'uptime attuale è vicino al 100% e non si prevedono interruzioni, il valore marginale del VPS è basso rispetto al costo ricorrente.
  • Non si ha familiarità con Linux server administration.
    Configurare systemd, nginx, TLS, firewall e monitoraggio richiede 4–8 ore iniziali e conoscenza del terminale Linux. Senza esperienza, il rischio di misconfigurazioni di sicurezza è elevato.
  • Il progetto è strettamente personale e non richiede mobilità.
    Se si accede al viewer sempre dalla stessa rete locale, il beneficio dell'accesso remoto è teorico, non pratico.
  • Subito anti-bot blocca l'IP del VPS.
    Gli IP dei datacenter europei (Hetzner, OVH) sono noti ai sistemi anti-bot. Subito potrebbe aumentare il tasso di blocco rispetto a un IP residenziale. Oggi il runner sul Mac Mini usa la connessione di casa.
  • Non si vuole investire tempo nella Mac App.
    Lo scenario C porta benefici solo se la Mac App viene sviluppata. Se l'obiettivo è solo la stabilità, lo scenario B è sufficiente e molto più rapido da implementare.

07 — Roadmap

Piano di migrazione suggerito (Scenario B)

Lo scenario B (VPS senza Mac App) è il percorso a minor rischio e massimo beneficio nel breve termine. Di seguito il piano operativo in 4 fasi per una migrazione completa in 1–2 settimane.

Fase 1
Ore 0–4

Provisioning VPS e ambiente

Creare VPS Hetzner CX22 (Ubuntu 24.04 LTS, €4.51/mese). Configurare SSH, firewall UFW, aggiornamenti automatici di sicurezza. Installare Python 3.11, pip, Playwright + Chromium in headless mode. Clonare il repository e installare requirements.lock.

Fase 2
Ore 4–8

Migrazione database e configurazione

Copia sicura di tracker.db (57 MB) sul VPS via rsync/scp. Aggiornare config.toml: playwright_channel = "chromium", host = "0.0.0.0", api_token = "". Configurare variabili d'ambiente per Telegram e Anthropic API key. Test locale degli scraper.

Fase 3
Giorni 2–5

Cron jobs + reverse proxy HTTPS

Sostituire i GitHub Actions workflow con crontab entries equivalenti. Configurare Caddy o nginx come reverse proxy con certificato TLS automatico via Let's Encrypt. Testare tutti e 6 i workflow per 48 ore in parallelo al Mac Mini.

Fase 4
Giorni 6–14

Switch-over e monitoraggio

Disabilitare i workflow GitHub Actions sul Mac Mini. Verificare l'uptime del VPS per 7 giorni. Configurare backup automatici con Hetzner Backup (€1/mese). Opzionale: registrare un dominio (es. trader.yourdomain.com) a €10–15/anno.

08 — Delibera

Domande per la decisione

Di seguito le domande chiave che determinano quale scenario è più adatto. La risposta a ciascuna orienta verso una scelta specifica. Rispondere a tutte prima di procedere.

D1. Il Mac Mini rimane acceso e connesso a internet 24/7 senza interruzioni pianificate?

Se il Mac è usato anche per altre attività (sviluppo, multimedia) che richiedono riavvii o spegnimento notturno, la continuità degli scraper è a rischio.

→ SÌ: Il Mac è stabile
→ NO: Si spegne spesso
SÌ → Il beneficio primario del VPS è ridotto. Valutare solo se si vuole accesso remoto. | NO → Il VPS è quasi obbligatorio per garantire la copertura dati.

D2. Si vuole accedere al viewer e alle notifiche da iPhone / rete mobile?

Oggi l'unico accesso è da browser locale. L'alert Telegram arriva, ma non è possibile esplorare i dati, filtrare o vedere il prezzo medio senza essere sul Mac.

→ SÌ: Accesso mobile importante
→ NO: Solo uso locale
SÌ → Scenario B o C fortemente raccomandato. | NO → Scenario A sufficiente nel breve termine.

D3. Si prevede di espandere il tracker ad altri mercati (PlayStation, Nintendo, Vinted) nei prossimi 12 mesi?

Aggiungere PlayStation 5 raddoppierebbe il volume di annunci Subito monitorati. Il carico computazionale di verify_sold (già 6.800+ annunci/sessione) crescerebbe proporzionalmente.

→ SÌ: Espansione prevista
→ NO: Solo Xbox
SÌ → Scenario B o C urgente: il Mac Mini diventerà insufficiente. Valutare CX32 (4 vCPU). | NO → Scenario A ancora sostenibile per 12–18 mesi.

D4. Si è disposti a investire 1–2 settimane di sviluppo per una Mac App nativa?

Una Mac App con WKWebView, menu bar e notifiche native richiederebbe circa 40–60 ore di sviluppo Swift. L'alternativa è accedere al viewer via browser Safari, che è già completamente responsive.

→ SÌ: Voglio l'app nativa
→ NO: Browser è sufficiente
SÌ → Scenario C. Procedere prima con il VPS (B), poi sviluppare l'app. | NO → Scenario B è completo e sufficiente senza sviluppo aggiuntivo.

D5. Si ritiene che Subito.it potrebbe bloccare gli scraper se girano su un IP di datacenter?

L'attuale runner usa un IP residenziale italiano (rete di casa). I VPS Hetzner hanno IP di datacenter tedesco, potenzialmente più soggetti a blocco da parte dei sistemi anti-bot di Subito. I dati di verify_sold mostrano già blocchi cffi (21% su alcune sessioni).

→ SÌ: Il rischio ban è reale
→ NO: Si può gestire
SÌ → Mantenere lo scraper Subito sul Mac Mini (IP residenziale) e migrare solo eBay, Fonti, Classify e il viewer sul VPS. Architettura ibrida. | NO → Migrazione completa al VPS è sicura, con monitoraggio del ban rate nelle prime 2 settimane.

D6. Si è a proprio agio con la gestione di un server Linux remoto (SSH, systemd, nginx)?

La migrazione richiede configurazione di: Ubuntu 24.04, firewall UFW, systemd services, Caddy/nginx con TLS, monitoraggio log. Senza esperienza precedente, stimare il doppio del tempo per ogni fase.

→ SÌ: Familiarità con Linux
→ NO: Poca esperienza server
SÌ → Procedere con Scenario B. Stima realistica: 4–8 ore totali. | NO → Considerare Railway.app, Render.com o Fly.io: deploy con git push, TLS automatico, zero configurazione server. Costo: €5–10/mese. Oppure rimandare al momento in cui si acquisisce più esperienza.
